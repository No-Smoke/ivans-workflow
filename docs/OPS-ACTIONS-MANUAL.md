## What It Does

The Ops Actions system tracks manual infrastructure tasks generated by the IWO pipeline — migrations, secrets, DNS changes, webhook configs, browser verifications. These are things Claude Code agents cannot do autonomously: running `wrangler d1 migrations apply`, setting API keys, configuring DNS records.

Every time IWO processes a handoff, it scans for these tasks automatically. New actions appear in the register, trigger phone notifications (critical/warning), and show on the kanban dashboard. The register lives at `ebatt/docs/agent-comms/.ops-actions.json`.

## Daily Workflow

**1. Check the dashboard.** Open `http://localhost:8787`. The banner at the top tells you the state at a glance:

- **Red banner** — critical ops actions are pending. Infrastructure work is needed before deploys can safely proceed.
- **Orange banner** — only warnings pending. Post-deploy integration tasks (webhooks, CRM config) that aren't blocking.
- **Green banner** — everything resolved. Pipeline is clear.

**2. Click "Ops Actions (N)" in the header** to open the `/ops` register page. Actions are grouped: pending critical first, then warning, then info, then completed and skipped (collapsed).

**3. Work through critical items.** Each action card shows:

- **Priority badge** (red/orange/blue) and **spec ID** — which spec generated it
- **Title and description** — what needs to be done
- **Category** (migration, secret, dns, config, etc.)
- **Age** — how long it's been pending
- **Verification command** (green monospace box) — click to copy, then run it to confirm the action is done
- **"POSSIBLY RESOLVED" badge** — stale detection flagged this because the source text disappeared from newer handoffs. Verify and close.

**4. Mark actions complete.** Currently requires editing `.ops-actions.json` directly or waiting for the CLI (Phase 4c). To mark complete manually:

```bash
# In the ebatt project root:
python3 -c "
import json, datetime
from pathlib import Path
p = Path('docs/agent-comms/.ops-actions.json')
d = json.loads(p.read_text())
for a in d['actions']:
    if a['id'] == 'ops-seed-027':  # <-- change this ID
        a['status'] = 'completed'
        a['resolved_by'] = 'vanya'
        a['resolved_at'] = datetime.datetime.now(datetime.timezone.utc).isoformat()
        a['notes'] = 'Stripe product created in LIVE dashboard'  # <-- add context
p.write_text(json.dumps(d, indent=2))
print('Done')
"
```

When `iwo ops complete <id>` ships (Phase 4c), this becomes: `iwo ops complete ops-seed-027 --notes "Stripe product created"`.

When running with `a` (auto-deploy-all) and `D` (auto-continue) enabled overnight:

- Critical ops actions still accumulate in the register but cannot block deploys (auto-deploy bypasses the gate).
- Check the dashboard in the morning. Red banner = infrastructure tasks were skipped overnight.
- The **smart deploy gate** (Phase 5, not yet implemented) will solve this properly — it auto-approves deploys only when relevant critical actions have been resolved.

**Recommended overnight sequence:**
1. Triage and resolve all critical ops actions before enabling `a` + `D`
2. Enable autonomous mode

## Understanding the Register

**Priorities** are auto-classified by keyword patterns:

- **Critical:** migrations, secrets, DNS records, "must run/create/configure", "not yet set/configured"
- **Warning:** webhook setup, config changes, CRM/Stripe integration
- **Info:** browser verifications, visual checks, end-to-end testing

**Categories:** migration, secret, dns, webhook, verification, config, email_infra, other

**Statuses:** pending → in_progress → completed | skipped

**Deduplication:** Actions are fingerprinted by `sha256(spec_id + normalized_text)`. The same action won't be added twice even if it appears in multiple handoffs for the same spec. However, semantically identical actions from different agents within the same spec can still create duplicates if the wording differs. See the duplication problem noted in "Current State" above.

**Stale detection:** When a newer handoff for a spec no longer mentions an action's source text, it gets a `stale_since` timestamp and the "POSSIBLY RESOLVED" badge. This means an agent may have fixed it. Verify with the verification command and mark complete.
